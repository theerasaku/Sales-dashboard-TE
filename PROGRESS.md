### 🐞 bug ที่เจอตอนรันจริง (แก้แล้ว)

หลังรัน `schema.sql` ทั้ง 5 ตารางตอบ **42501 permission denied** ไม่ใช่ `[]`
→ project Supabase รุ่นใหม่ **ไม่ GRANT ให้ `authenticated` อัตโนมัติ** แบบรุ่นเก่า

RLS คุมแค่ "เห็นแถวไหน" ส่วน GRANT คุม "แตะตารางได้ไหม" — ขาดตัวใดตัวหนึ่งก็ใช้งานไม่ได้
ถ้าไม่เจอตรงนี้ อาการที่จะเกิดคือ **step 1.2 ล็อกอินผ่านแต่หน้าจอว่างเปล่า**

เพิ่มใน `policies.sql` แล้ว:
```sql
grant usage on schema public to anon, authenticated;
grant select, insert, update, delete on <5 ตาราง> to authenticated;
```

### 🐞 bug ที่ 2: ตั้ง admin คนแรกไม่ได้ (แก้แล้ว)

trigger `guard_profile_privilege` เช็กแค่ `is_admin()` → ใน SQL Editor `auth.uid()` เป็น null
→ `is_admin()` = false → trigger บล็อก เกิดปัญหาไก่กับไข่ตั้ง admin คนแรกไม่ได้

แก้เป็น `if auth.uid() is null or is_admin()` — auth.uid() null = มาจาก SQL Editor /
service key ซึ่งเชื่อถือได้ ส่วนคนยิงผ่าน REST API มี auth.uid() เสมอ จึงยังบล็อกอยู่

### 🐞 bug ที่ 3: ข้อความไทยใน DB พัง (แก้แล้ว)

`LANG` ในเครื่องว่าง → `pbcopy` แปลง UTF-8 เป็น MacRoman → ไทยเพี้ยนตั้งแต่ clipboard
ตรวจไม่เจอตอนแรกเพราะใช้ `pbcopy | pbpaste` ซึ่งแปลงกลับด้วยวิธีเดียวกัน

**กติกาใหม่:** ใช้ `LC_CTYPE=UTF-8 pbcopy` เสมอ · ตรวจด้วย `osascript -e 'the clipboard as text'`
(บันทึกไว้ใน CLAUDE.md แล้ว)

## ✅ 1.3 F2 Data Adapter (21 ก.ค. 2026)

เติม query จริงใน `supabase-adapter.js` + ทำ `local-adapter.js` ให้ผลลัพธ์ตรงกัน

| เมธอด | ทำอะไร |
|---|---|
| `listTeams()` | 5 ทีมที่ active |
| `listPending(opt)` | กรอง activeOnly/team/stage/**ช่วงเดือนคาดปิด**/ค้นหา + เรียง + limit |
| `getPending(id)` | งาน + `follow_logs` + `project_contacts` ยิงครั้งเดียว |
| `savePending(row)` | มี id = PATCH · ไม่มี = POST · ใส่ created_by/updated_by ให้อัตโนมัติ |
| `archivePending(id, on)` | **ทางลบปกติของ sale** — set is_active=false |
| `deletePending(id)` | ลบถาวร (RLS ปล่อยเฉพาะ admin) |
| `listFollowLogs` / `addFollowLog` | บันทึกติดตาม |

`listCustomers` / `listActivities` / `getDashboardStats` ยังเป็น `notReady()` ตามเดิม —
ตาราง customers/activities สร้างใน 2.1 · views ใน 1.5 **ไม่ใช่ของค้าง แต่ยังไม่ถึงคิว**

**สิ่งที่กันไว้ในโค้ด**
- `SORTABLE` whitelist — ไม่เอาชื่อคอลัมน์จาก UI ไปต่อ URL ตรง ๆ
- `READONLY` — ตัด `created_at`/`teams`/`follow_logs` ทิ้งก่อน PATCH ไม่งั้น PostgREST ตอบ 400
- `safeSearch()` — ล้าง `,()*\` ที่เป็นอักขระพิเศษของ PostgREST
- ช่องว่างเปล่า → `null` (ไม่งั้นคอลัมน์ date ที่ปล่อยว่างจะ error)
- `nullslast` ทุก order — งานที่ยังไม่กรอกวันที่ต้องอยู่ท้าย ไม่ใช่ลอยขึ้นหัวตาราง

**ผลทดสอบ**
- `node --check` ผ่านทั้ง 4 ไฟล์
- เช็ก parity: facade เรียก 17 เมธอด → มีครบทั้ง 2 adapter
- ยิง query จริงของ `listPending` ใส่ live DB → `401 / 42501 permission denied for anon`
  พิสูจน์ 2 อย่าง: **anon อ่านไม่ได้จริง** และ **PostgREST แปล query ผ่าน**

### บั๊กที่เจอหลังล็อกอินจริงครั้งแรก (21 ก.ค. 2026)

**1. ป้ายชื่อเมนูตกบรรทัดบน iPad** — CSS เขียนว่า `.nav-item span:not(.ico) { display:none }`
เพื่อซ่อนป้ายชื่อตอนแถบข้างแคบเหลือ 68px **แต่ใน `index.html` ป้ายชื่อเป็น text เปล่า ไม่ได้อยู่ใน `<span>`**
→ selector ไม่ตรงกับอะไรเลย ตัวหนังสือจึงยังอยู่แล้วตกบรรทัดในแถบ 68px

วัดที่ 1016px: ปุ่มเมนูสูง **41 / 62 / 83px ไม่เท่ากัน** (ตกบรรทัด 1–3 บรรทัด) เนื้อหาล้นถึง 62px
แก้โดยครอบป้ายชื่อด้วย `<span class="lbl">` + เปลี่ยน selector เป็น `.nav-item .lbl`
หลังแก้: **สูงเท่ากันหมด 40px** เนื้อหากว้างสุด 51px พอดีในแถบ 68px

> บทเรียน: selector ที่ "ไม่ตรงกับอะไรเลย" ไม่มี error ไม่มี warning — เงียบสนิท
> เขียน CSS ซ่อนอะไรต้องเช็กด้วยว่า markup มี hook ให้เกาะจริง

**2. หน้าภาพรวมขึ้น "โหลดหน้านี้ไม่สำเร็จ"** — `getDashboardStats()` โยน error ทำให้พังทั้งหน้า
และข้อความบอกผิดเฟส ("มาใน Phase 1.3" ทั้งที่ dashboard views อยู่ 1.5)
แก้ 3 จุด:
- `getDashboardStats()` นับ pending ได้จริงแล้วผ่าน `Prefer: count=exact` (ไม่โหลดข้อมูลลงมือถือ)
- ตัวที่ยังไม่มีตารางคืน `null` → UI แสดง **"—"** (คนละความหมายกับ `0` ที่แปลว่านับแล้วไม่มี)
- `dashboard.js` ครอบ try/catch — ต่อ DB ไม่ได้ก็ยังเห็นหน้าจอ ไม่ใช่จอว่าง
- `notReady()` รับพารามิเตอร์เฟส ไม่ hardcode เลขผิดอีก

> ⚠️ ระหว่างทดสอบเจอไทยเพี้ยนใน output — **ตรวจแล้วเป็นหน้าทดสอบของผู้ช่วยเองที่ลืมใส่

> `docs/index.html` ตัวจริงมี charset ครบ ไม่กระทบ — พิสูจน์โดยใส่ charset แล้วรันซ้ำ ไทยออกถูกทันที

## ✅ 1.4b บันทึกความคืบหน้ารายวัน — ทำให้เห็นและใช้ง่าย (22 ก.ค. 2026)

**ข้อติจากการใช้งานจริง:** "โครงสร้างข้อมูล 1 แถว ยังไม่มีรายบันทึกความคืบหน้าแต่ละวัน"

ตรวจแล้ว — ตาราง `follow_logs` และ UI มีอยู่ตั้งแต่ 1.4 **แต่ถูกซ่อน 2 ชั้น**
1. ส่วนบันทึกโผล่เฉพาะตอนเปิดงานเดิมมาแก้ — ตอน "เพิ่มงานใหม่" ไม่มีเลย ผู้ใช้จึงไม่เคยเห็น
2. จากหน้าตารางมองไม่ออกว่างานไหนมีบันทึกอะไร ต้องเปิดทีละงาน

แก้ 3 อย่าง:
- **คอลัมน์ "ความคืบหน้าล่าสุด"** ในตาราง + แสดงในการ์ดมือถือ (วันที่ · ผู้บันทึก · ข้อความ)
  ฝั่ง DB ใช้ embedded query `follow_logs.order=log_date.desc&follow_logs.limit=1`
  → ดึงมาแค่อันล่าสุดของแต่ละงาน ไม่ลากทั้งประวัติลงมือถือ
- **ปุ่ม `＋ บันทึก` ในทุกแถว** → เปิดแผงเล็ก 4 ช่องตามฟอร์มกระดาษ (DATE/BY/RESPONSE/NEXT DOING)
  พร้อมประวัติย้อนหลัง — **งานประจำวันไม่ต้องเปิดฟอร์ม 42 ช่องอีกต่อไป**
- ตอนสร้างงานใหม่ **แสดงหัวข้อบันทึกติดตามพร้อมคำอธิบาย** แทนที่จะซ่อนหายไปเฉย ๆ

### 🐛 บั๊กที่เจอระหว่างทำ: ปุ่มซ้อนในปุ่ม

การ์ดมือถือเป็น `<button>` พอใส่ปุ่ม `＋ บันทึก` เข้าไปข้างใน กลายเป็น `<button>` ซ้อน `<button>`
ซึ่ง **HTML ไม่อนุญาต → parser ดีดปุ่มในออกมานอกการ์ด โครงสร้างการ์ดพัง**

แก้เป็น `<div class="pcard" role="button" tabindex="0">` + รับ Enter/Space เองเพื่อคงการใช้คีย์บอร์ด

> บทเรียน: ปุ่มซ้อนปุ่มไม่มี error ไม่มี warning — DOM เงียบ ๆ ผิดจาก HTML ที่เขียน
> ต้องเทสต์ว่า element ที่ควรอยู่ข้างใน **ยังอยู่ข้างในจริง** ไม่ใช่แค่ว่ามันมีอยู่

**ผลทดสอบ** — 2 ชุด รวม 30 เคส ผ่านหมด
- ชุดบันทึกความคืบหน้า 20 เคส: สร้างงาน → ปุ่มบันทึกเร็ว → กันบันทึกเปล่า → บันทึก 2 ครั้งต่างวัน →
  ตารางโชว์อันล่าสุด ไม่ใช่อันเก่า → เก็บครบ 2 รายการไม่ทับกัน → ฟอร์มเต็มเห็นครบ → XSS escape
- ชุด regression 10 เคส: ปุ่มอยู่ในการ์ดจริง · เรียง · ซ่อน/แสดงคอลัมน์ · ฟอร์มยังครบ 42 ช่อง · archive

## ✅ 1.4c ปุ่มแก้ไขบันทึกที่เขียนไปแล้ว (22 ก.ค. 2026)

## ✅ 1.4d แก้ข้อมูลหายตอนบันทึก (22 ก.ค. 2026)

**ข้อติจากการใช้งานจริง:** พิมพ์บันทึกติดตามแล้วกดปุ่ม **บันทึก** (ม่วง) โดยไม่ได้กด
"+ เพิ่มบันทึก" → ข้อความหายเงียบ ๆ ไม่มีคำเตือน

ตรวจแล้วเป็นปัญหาเดียวกัน **2 ทิศทาง** — เจ้าของเจอทิศเดียว แต่อีกทิศร้ายกว่า:

| ทิศ | อาการ | ความเสียหาย |
|---|---|---|
| 1 | พิมพ์บันทึก → กด "บันทึก" | บันทึก 4 ช่องหาย |
| 2 | แก้ 42 ช่องค้างไว้ → กด "+ เพิ่มบันทึก" | **ที่แก้ทั้ง 42 ช่องหาย** |

ทิศที่ 2 เกิดเพราะ `lgAdd` เรียก `openDetail()` ใหม่ทั้งแผง → ดึงข้อมูลจาก DB มาทับสิ่งที่พิมพ์ค้างไว้
(ปัญหาเดียวกันเกิดกับปุ่มแก้ไขบันทึกเก่าด้วย ทั้งในฟอร์มเต็มและแผงเล็ก)

**แก้ 3 อย่าง**

### 🐛 1.4g แก้ปฏิทินเลื่อนเดือน/ปีไม่ได้ (22 ก.ค. 2026)

**ข้อติจากการใช้งานจริง:** "กดปุ่มเลื่อนเดือน เลื่อนปี ไม่ได้เลย ปฏิทินถูกล็อกไว้แค่เดือนปัจจุบัน"

ทดสอบเดิม **ผ่านหมดทั้งที่ของจริงพัง** เพราะเทสต์ยิง event ตรง ๆ
(`el.dispatchEvent(new Event('change'))`) ไม่ได้จำลองการกดจริง
พอจำลองครบลำดับ `pointerdown → mousedown → pointerup → mouseup → click` ก็เจอทันที **2 บั๊กคนละตัว**

**บั๊ก 1 — `<select>` เปิดดรอปดาวน์ไม่ได้**
`pop.addEventListener('mousedown', e => e.preventDefault())` ที่ใส่ไว้กันช่องกรอกเสีย focus
**ไปบล็อกพฤติกรรมปกติของ `<select>`** ทำให้ดรอปดาวน์เดือน/ปีไม่เปิด
→ ลบทิ้ง (ช่องกรอกเป็น `readonly` อยู่แล้ว ไม่ต้องหวง focus)

**บั๊ก 2 — กด ‹ › แล้วปฏิทินปิดทันที**
ปุ่มนี้ทำให้ปฏิทินวาดใหม่ (`pop.innerHTML = …`) → **ปุ่มที่กดหลุดจาก DOM**
พอ event วิ่งต่อไปถึงตัวดักคลิกนอกพื้นที่ `e.target.closest('.dp-pop')` หาไม่เจอ (เพราะ target ลอยแล้ว)
เลยสั่งปิดทั้งที่ผู้ใช้กดข้างใน → ดูเหมือน "ล็อกอยู่เดือนเดียว"
→ เปลี่ยนไปใช้ **`e.composedPath()`** ซึ่งเก็บเส้นทางไว้ตั้งแต่ตอน dispatch ไม่โดนผลจากการวาดใหม่

> บทเรียนสำคัญ: **เทสต์ที่ยิง event สังเคราะห์ไม่พิสูจน์ว่าคนกดได้จริง**
> ส่วนที่ผู้ใช้ต้อง "กด" ต้องจำลองลำดับเมาส์ให้ครบ ไม่งั้น `preventDefault` ที่บล็อกของจำเป็นจะรอดสายตา
> และ **การวาด DOM ใหม่ระหว่างจัดการ event ทำให้ `closest()` ของ listener ถัดไปพัง** — ใช้ `composedPath()` แทน

**ผลทดสอบ 26 เคส ผ่านหมด** (คลิกจริงครบลำดับทุกเคส)
- กด › ‹ ต่อเนื่องหลายครั้ง · **ข้ามสิ้นปี → ม.ค. ปีถัดไป** · **ข้ามต้นปี → ธ.ค. ปีก่อน**

ทดสอบทั้งบนหน้าเปล่าและ**ในแอปจริงผ่านหน้า login → แท็บ Pending → เปิดงาน → กดปฏิทิน**

ผลคือ **ทำงานถูกต้องทุกขั้น** และ `curl` ไฟล์บนเว็บจริงยืนยันว่าเป็นเวอร์ชันที่แก้แล้ว
→ สรุปได้ว่า **เบราว์เซอร์ของผู้ใช้ยังรันโค้ดเก่า**

**ต้นเหตุ:** GitHub Pages ส่ง `cache-control: max-age=600`
เบราว์เซอร์จึงเสิร์ฟไฟล์เก่าจาก HTTP cache ได้นานถึง 10 นาที
service worker เป็น network-first ก็ไม่ช่วย เพราะ `fetch()` ของมันก็ไปหยิบจาก HTTP cache เหมือนกัน

**แก้ 2 อย่าง ที่ต้นเหตุ ไม่ใช่บอกให้ผู้ใช้ล้าง cache**
1. `sw.js` ใช้ `fetch(req, { cache: 'no-cache' })` สำหรับไฟล์ `.js/.css/.html`
   → บังคับตรวจกับเซิร์ฟเวอร์ทุกครั้ง (ยังใช้ ETag ถ้าไม่เปลี่ยนตอบ 304 เบามาก)
   **deploy แก้บั๊กแล้วผู้ใช้ได้ของใหม่ทันที**
2. **แสดงเลขเวอร์ชันบนหน้าจอ** (ป้ายข้าง "ต่อฐานข้อมูลแล้ว" + มุมล่างซ้าย)
   → ต่อไปถ้าเจอบั๊ก ดูเลขก็รู้ทันทีว่ารันโค้ดเวอร์ชันไหน ไม่ต้องเดากันอีก

> บทเรียน 2 ชั้น:
> 1. **`dispatchEvent` ไม่ใช่การคลิกจริง** — ต้องใช้ CDP/puppeteer เมื่อทดสอบสิ่งที่ผู้ใช้ต้องกด
> 2. **ถ้าแก้แล้วผู้ใช้ยังเจอบั๊กเดิม ให้สงสัย cache ก่อนสงสัยตรรกะ** — และวิธียืนยันคือ
>    ทำให้เวอร์ชันมองเห็นได้ ไม่ใช่ถามว่า "ล้าง cache หรือยัง"
