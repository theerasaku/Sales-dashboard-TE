# WishtoHave — ฟังก์ชันที่อยากได้ (ยังไม่อยู่ใน roadmap)

> **ไฟล์นี้เจ้าของโปรเจกต์เขียน** — คิดอะไรออกระหว่างใช้งาน/พัฒนา จดทิ้งไว้ตรงนี้ก่อน
> ไม่ต้องคิดว่าทำได้ไหม ทำตอนไหน แพงไหม — **จดดิบ ๆ ก็พอ Claude จะประเมินให้ทีหลัง**

## วิธีใช้

**คุณทำ:** พิมพ์บรรทัดใหม่ใต้ `📥 กล่องรับ` — บรรทัดเดียวก็พอ ไม่ต้องจัดรูปแบบ

**บอก Claude ว่า:** `"อ่าน WishtoHave ให้หน่อย"` → Claude จะประเมินทีละข้อ แล้วย้ายลงไปที่ `✅ ประเมินแล้ว` พร้อมคำตอบ 5 อย่าง:

| หัวข้อ | ตอบอะไร |
|---|---|
| **ควรทำ step ไหน** | เลขที่ตรงกับ roadmap หรือ "step ใหม่" ถ้าไม่มีที่ลง |
| **กระทบโครงสร้างไหม** | แตะ schema / adapter / RLS หรือแค่ UI ชั้นนอก |
| **ต้อง migration ไหม** | ถ้าตารางมีข้อมูลจริงแล้วต้องย้ายข้อมูลด้วยหรือเปล่า |
| **⭐ ราคาถ้าเลื่อน** | ทำตอนนี้ vs ทำทีหลัง ต่างกันแค่ไหน — **ข้อสำคัญที่สุด** |
| **ติดอะไรก่อน** | ต้องรอ step ไหนเสร็จก่อนถึงทำได้ |

**ทำไม "ราคาถ้าเลื่อน" ถึงสำคัญที่สุด:** ของบางอย่างถ้าเติมตอนตารางยังว่างคือฟรี แต่ถ้ารอจนทีมกรอกข้อมูลจริงไปแล้ว
กลายเป็นงาน migration + เสี่ยงข้อมูลเพี้ยน — ต้นทุนต่างกันหลายเท่าทั้งที่เป็นฟีเจอร์เดียวกัน
(เคสจริงในโปรเจกต์นี้: `is_active`/`archived_at` ใส่ไว้ตั้งแต่ step 1.1 → step 2.5 Archive เลยไม่ต้อง migration เลย)

## ✅ ประเมินแล้ว

<!-- Claude ย้ายข้อจากกล่องรับมาไว้ตรงนี้ พร้อมผลประเมิน -->

### ปุ่มลบบันทึกติดตาม
_จดเมื่อ: step 1.4c (22 ก.ค. 2569) · ประเมินเมื่อ: 22 ก.ค. 2569_

ตอนนี้บันทึกติดตามแก้ไขได้แต่ลบไม่ได้ — อยากลบบันทึกที่จดผิดทิ้ง

| หัวข้อ | ผลประเมิน |
|---|---|
| ควรทำ step ไหน | **2.6** (พร้อมหัวหน้าเซ็นรับทราบ) |
| กระทบโครงสร้างไหม | ไม่กระทบ schema — เพิ่ม policy `follow_delete` + ปุ่มใน UI |
| ต้อง migration ไหม | ไม่ต้อง |
| ⭐ ราคาถ้าเลื่อน | **ถูกลงถ้ารอ** — ทำตอนนี้ต้องกลับมาแก้ซ้ำตอน 2.6 อยู่ดี |
| ติดอะไรก่อน | step 2.4 (ต้องมี role `manager` ก่อน) |

**เหตุผลที่รอ:** ระบบนี้ทำแค่ backup ไม่มี rollback รายแถว — ลบแล้วกู้ไม่ได้
พอถึง 2.6 จะมี `signoffs` มาล็อกว่า **บันทึกที่หัวหน้าเซ็นรับทราบไปแล้วห้ามลบ**
ถ้าทำปุ่มลบตอนนี้ = เปิดช่องให้ลบหลักฐานที่หัวหน้าตรวจไปแล้ว แล้วต้องกลับมาปิดช่องทีหลัง
